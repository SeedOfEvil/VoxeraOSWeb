What works now

Daily-driver baseline features

  • Cloud mission planner via voxera missions plan "<goal>" with safety gates intact
  • Deterministic simple-write planning for note/file goals using a single files.write_text step
  • Queue daemon + approval inbox under pending/approvals/*.approval.json
  • Queue status UX + panel insights for operators
  • DEV-only auto-approve for strict allowlist (system.settings)
  • Human-friendly inbox commands and queue lifecycle actions
  • Update tooling and systemd user service management

Architecture Layers

  1. Substrate OS — Ubuntu/Fedora/etc. (drivers, filesystem, networking)
  2. AI Control Plane — intent routing, planning, memory, tools, policy
  3. Experience Layer — voice shell + confirmation panel, later richer UI modes

Why it feels different

Trust-first operator experience

No silent risky changes

Risky actions (network/install/credentials) remain approval-gated instead of disappearing into agent magic.

Queue intake contract

Only notes/queue/inbox/*.json is supported for primary job drops. Legacy drops are auto-relocated and audited.

Artifact-first debugging

Done, pending, and failed jobs all emit consistent artifact bundles for timeline + stdout/stderr inspection.
