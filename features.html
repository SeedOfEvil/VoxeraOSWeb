Feature Overview

Alpha v0.1.4 capabilities that make Voxera reliable

This release is not a flashy promise build. It is a stability + UX baseline focused on queue reliability, approval workflows, deterministic planning paths, and operator visibility.

Mission Planning

  • voxera missions plan "<goal>" uses cloud planning while preserving local policy + approval gating.
  • Deterministic fast-path for common note/file writes emits exactly one files.write_text step.
  • Planner rewrites sandbox-host-inappropriate steps to safer manual confirmation clipboard actions when verification intent is not explicit.
  • Built-in missions + repo mission files + user mission files with deterministic resolution order.

Queue & Inbox Operations

  • Queue daemon processes mission/goal JSON jobs via inbox intake contract.
  • Approval artifacts written to pending/approvals/*.approval.json.
  • Commands: status, approvals list/approve/deny, cancel, retry, pause, resume.
  • Human-friendly queueing: voxera inbox add and voxera inbox list.

Artifacts, Failures, and Retention

  • Per-job artifact bundles under notes/queue/artifacts/<job_id>/ with plan, actions timeline, stdout/stderr, generated files metadata.
  • Failed-job sidecar contract: failed/<job_stem>.error.json with schema versioning and validation.
  • Retention policies by age/count with prune summaries and health counters.
  • Invalid sidecars are ignored in snapshots and audited deterministically.

Services, Panel, and Updates

  • Systemd user service lifecycle via Make targets (services-install/status/restart/stop/disable).
  • Panel supports queue + mission creation and insights; POST mutation endpoints by default.
  • make update / scripts/update.sh restart services if installed and enabled.
  • Update flow avoids deleting anything under ~/VoxeraOS/notes.

Security & Safety Model

Guardrails by design

Audited actions

Everything records what happened, why it happened, and how to undo it where possible.

Skill-declared permissions

Skills declare scope/needs; policies decide allow/ask/deny.

Sandbox isolation

Rootless Podman backend, read-only root, writable /work only, and default network disabled.
